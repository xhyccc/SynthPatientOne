# MDT Builder

## Overview

Given a patient profile folder `{patient_id}_profile/` (produced by `PatientProfileBuilder`),
this skill runs a full Multidisciplinary Team review. The LLM acts as the MDT Coordinator,
simulating each department's independent review, facilitating the meeting discussion,
resolving inter-departmental conflicts, and producing a consensus report with actionable follow-up.

### What the Pipeline Produces

```
{patient_id}_profile/mdt/
├── pre_meeting/
│   ├── agenda.md                     # MDT scope, core question, participating departments
│   ├── departmental_reviews/         # Independent review per department (LLM)
│   │   ├── 1.1_cardiology.md
│   │   ├── 7.1_radiology.md
│   │   └── ... (only departments with MDT-relevant findings)
│   └── evidence_summary.md           # Aggregate of all objective evidence
├── meeting/
│   ├── minutes.md                    # Full meeting record with positions & discussion
│   └── conflicts_resolution.md       # Structured conflict resolution log
├── consensus_report.html             # Final MDT consensus — self-contained HTML
├── patient_summary.md                # Plain-language patient-facing summary
├── follow_up_plan.md                 # Timeline, triggers, responsible departments
└── quality_audit.md                  # Process quality self-assessment
```

### Core Principle

**The MDT does not re-extract data.** It reads the audited status tables, history tables,
and profile.html from `PatientProfileBuilder`. Every recommendation cites these as evidence.
The MDT's role is to interpret, synthesize, and decide — not to re-audit the raw records.

---

## When to Use This Skill

- A complex patient requires coordinated input from multiple specialties.
- Treatment decisions involve conflicting departmental recommendations.
- A formal MDT consensus report is needed for the medical record.
- Clinical governance requires documented multidisciplinary decision-making.
- Tumor board, complex trauma board, transplant evaluation, or similar MDT settings.

## Pipeline Phases

### Phase 1 — TEAM FORMATION (LLM-driven)

1. Read the patient profile (`profile.html`, `status_tables/`, `history_tables/`).
2. Identify the **MDT scope** based on key diagnoses and flagged unknowns.
3. Determine **which departments must participate** — at minimum: the department with the primary diagnosis, Radiology (if imaging exists), Pathology (if biopsies exist), and any department with critical unknowns.
4. Designate the **Lead Clinician (Chair)** — typically the referring physician's department.
5. Create `pre_meeting/agenda.md` with the core clinical question.

### Phase 2 — PRE-MEETING (LLM-driven)

For each participating department:

1. **Read** the department's status table and history table from the profile.
2. **Write** a `departmental_reviews/{dept}.md` following the template in `references/mdt_output_spec.md`:
   - Findings summary (from status/history tables)
   - Assessment (clinical interpretation)
   - Recommendations (specific, actionable, evidence-cited)
   - Concerns / Contraindications (any reason to advise AGAINST interventions)
   - Data Gaps (missing data preventing confident recommendation)
3. **Aggregate** all objective evidence (imaging, pathology, labs, exams) into `evidence_summary.md`.

**Critical rule:** Every statement in a departmental review MUST cite the source status or history table via relative path.

### Phase 3 — MEETING (LLM-driven)

Simulate the MDT meeting:

1. **Case Presentation:** The chair (or referring department) presents the patient summary and core clinical question.
2. **Objective Data Review:** Radiology and Pathology findings are presented as undisputed facts.
3. **Departmental Positions:** Each department states its assessment and recommendation. Disagreements are recorded, not suppressed.
4. **Conflict Resolution:** When departments disagree, apply the resolution framework:
   - Consult clinical guidelines (NCCN, ESC, AHA, etc.)
   - Compare morbidity/mortality risks of each option
   - Assess patient factors (ECOG, comorbidities, preferences)
   - Chair makes the final decision with full rationale
5. **Consensus Outcome:** Select one of:
   - Therapeutic Intervention (specify modality, timeline, responsible department)
   - Further Diagnostics Required (specify tests, urgency)
   - Active Surveillance / Monitoring (define interval, escalation triggers)
   - Referral to External Specialist
   - Palliative / Supportive Care
6. Write `meeting/minutes.md` and `meeting/conflicts_resolution.md` (if applicable).

### Phase 4 — POST-MEETING (LLM-driven)

1. **Consensus Report:** Write `consensus_report.html` — self-contained HTML with executive summary, evidence review, departmental positions, conflict resolution, consensus decision, implementation plan.
2. **Patient Summary:** Write `patient_summary.md` in plain language (no medical jargon) for the patient and family.
3. **Follow-Up Plan:** Write `follow_up_plan.md` with immediate actions, short-term follow-up, MDT re-evaluation date, and outcome tracking metrics.

### Phase 5 — IMPROVEMENT (LLM-driven)

1. **Quality Audit:** Self-assess the MDT process against quality indicators.
2. Write `quality_audit.md` with a completed indicators table and process improvement notes.

## Conflict Resolution Framework

When two departments disagree, the LLM applies this structured framework:

```
1. IDENTIFY the exact point of disagreement.
2. EVIDENCE: What does each department cite?
3. RISK COMPARISON: Compare morbidity, mortality, QoL impact.
4. PATIENT FACTORS: ECOG, comorbidities, preferences, age.
5. GUIDELINE REFERENCE: Cite specific guideline recommendation.
6. CHAIR DECISION: Recommendation with full rationale.
```

## How to Use

When this skill is activated, the LLM executes all 5 phases directly — **no scripts, no external tools**:

1. **Read** the patient profile at `{patient_id}_profile/` (status_tables, history_tables, profile.html, logs).
2. **Create** the `{patient_id}_profile/mdt/` folder and all subdirectories (`pre_meeting/departmental_reviews/`, `meeting/`).
3. **Execute** Phases 1–5 as described above, writing every output file directly.
4. **Cite** all evidence using relative paths to status/history tables.

> This skill is **pure LLM reasoning**. The MDT protocol is a cognitive workflow — reading, interpreting, simulating, deciding, documenting. There is no file conversion, no OCR, no API integration. Every file is written directly by the LLM.

## Best Practices

1. **Do not re-extract data.** The MDT reads the profile. It does not re-read raw records.
2. **Every recommendation cites evidence.** Use relative paths to status/history tables.
3. **Record dissenting opinions.** Consensus does not mean unanimity. Document dissents.
4. **Patient-centered.** When clinical evidence is equivocal, patient factors decide.
5. **Actionable outputs.** Every consensus decision includes WHO does WHAT by WHEN.
6. **Guideline-referenced.** Cite specific clinical guidelines, not vague "standard of care."
7. **Complete the quality audit.** Self-assessment is mandatory, not optional.

## Limitations

- MDT simulation is based on documented data only. Missing data in the profile = missing data in MDT.
- The LLM simulates department perspectives; it does not replace actual specialist consultation.
- Clinical guidelines cited should be verified by a human clinician.
- This skill organizes decision-making; it does not provide medical treatment.
